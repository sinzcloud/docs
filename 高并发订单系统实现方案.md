# 高并发订单系统实现方案（项目落地版）

> 目标：在高 QPS 场景下实现“下单快、库存准、支付一致、查询抗压”的订单系统。

---

## 1. 总体架构

典型链路：

用户请求 → CDN → Nginx/LB → API网关(鉴权/限流) → 订单服务集群  
→ Redis(缓存/库存/幂等) → MQ(削峰) → MySQL(分库分表)  
→ 监控系统(Prometheus/Grafana) + 日志系统(ELK/Loki)

核心原则：

- 服务无状态（可水平扩容）
- 热点读写尽量走 Redis
- 写操作异步化（MQ削峰）
- 数据一致性用最终一致性（补偿/重试）

---

## 2. 服务拆分（推荐）

- **Order Service**：订单创建、订单状态流转、订单查询
- **Inventory Service**：库存预扣/回滚/落库
- **Payment Service**：支付回调、退款、支付流水
- **Product Service**：商品信息、价格、SKU
- **User Service**：用户地址、收货信息
- **Notify Service**：短信/邮件/站内信
- **MQ Consumer**：订单创建后异步任务处理

---

## 3. 订单状态机设计

订单状态必须固定并控制流转：

- `CREATED`：订单已创建待支付
- `PAID`：已支付
- `CANCELLED`：已取消
- `SHIPPED`：已发货
- `FINISHED`：已完成
- `REFUNDING`：退款中
- `REFUNDED`：已退款

合法流转：

- CREATED → PAID
- CREATED → CANCELLED
- PAID → SHIPPED
- SHIPPED → FINISHED
- PAID → REFUNDING → REFUNDED

避免出现 CANCELLED → PAID 等非法状态。

## 5. 订单号生成（分布式唯一ID）

不建议使用数据库自增ID（分库分表冲突）。

推荐：

- Snowflake 雪花算法
- 美团 Leaf
- Redis INCR + 日期前缀

订单号最好包含时间便于排序，例如：

`202604031234567890`

---

## 6. 高并发下单流程（核心）

### 6.1 下单同步链路（必须短）

1. 校验登录态、参数合法性
2. 校验商品是否可售、价格是否有效
3. 幂等校验（token/request_id）
4. Redis 预扣库存（原子扣减）
5. 创建订单（写 MySQL）
6. 发送 MQ 消息（订单创建事件）
7. 返回下单成功（订单号、支付信息）

### 6.2 下单后异步链路（MQ消费）

- 扣减库存落库
- 清空购物车
- 增加销量
- 发短信/站内信
- 写操作日志

---

## 7. 库存防超卖实现

### 7.1 推荐方案：Redis 预扣库存 + 异步落库

库存数据提前写入 Redis：

key: `stock:sku:{skuId}`

下单时通过 Lua 脚本原子扣减：

```lua
if redis.call('get', KEYS[1]) == false then
  return -2
end
local stock = tonumber(redis.call('get', KEYS[1]))
if stock <= 0 then
  return -1
end
redis.call('decr', KEYS[1])
return 1
```

返回值说明：

- `1` 扣减成功
- `-1` 库存不足
- `-2` key不存在

成功后发 MQ：`ORDER_CREATED`

消费者异步扣减 MySQL 库存。

## 8. 幂等设计（防止重复下单）

高并发下必须考虑用户重复点击/重试。

### 8.1 Token 幂等（推荐）

流程：

- 前端请求 `/order/token`
- 后端生成 token，写入 Redis（一次性）
- 下单时携带 token
- 服务端校验 token 是否存在，校验成功后删除

Redis：

- key: `order:token:{userId}`
- value: token
- TTL：60秒

---

### 8.2 request_id 幂等（备选）

每次请求带 request_id：

- order_main 表新增 request_id 字段
- unique(request_id)

重复请求直接返回同一个订单。

---

## 9. 订单超时关闭（关单+库存回滚）

订单创建后若 15 分钟未支付：

- 自动取消订单
- 回滚 Redis 库存
- 回滚 MySQL 库存（如果已落库）

### 推荐实现：MQ 延迟消息

订单创建时发送延迟消息：

- topic: `ORDER_TIMEOUT_CHECK`
- delay: 15分钟

消费者处理逻辑：

- 查询订单状态
- 若仍为 CREATED → CANCELLED
- 回滚库存并发送取消事件

---

## 10. 支付回调一致性（关键）

支付回调常见问题：

- 回调重复
- 回调延迟
- 回调顺序错乱

必须做到幂等。

### 支付回调流程

1. 校验签名
2. 根据 order_no 查询订单
3. 如果订单已是 PAID，直接返回 success（幂等）
4. 更新订单状态为 PAID
5. 写 pay_record
6. 发送 MQ：`ORDER_PAID`

---

## 11. 分库分表策略（订单系统必做）

订单量大后必须分库分表。

推荐按 user_id hash：

- order_main_00 ~ order_main_63
- order_item_00 ~ order_item_63

同一个 user_id 的订单落在固定分片，查询效率高。

---

## 12. 查询性能优化（Redis + ES）

订单查询属于高频读。

### 12.1 Redis 缓存订单详情

- key: `order:detail:{orderNo}`
- TTL：5~30分钟

热点订单（刚下单）查询量大，缓存收益极高。

### 12.2 用户订单列表

如果订单列表查询压力大：

- 使用 ES 建立订单索引（支持分页、搜索）
- 或按 user_id 分表保证查询稳定

## 13. 消息队列事件设计

订单系统常见事件：

- `ORDER_CREATED`
- `ORDER_PAID`
- `ORDER_CANCELLED`
- `ORDER_TIMEOUT`
- `ORDER_SHIPPED`
- `ORDER_REFUNDING`
- `ORDER_REFUNDED`

消费者任务：

- 库存落库/回滚
- 销量统计
- 清购物车
- 通知服务
- 积分系统

---

## 14. 分布式事务处理方式

不推荐传统 2PC。

推荐：**最终一致性（MQ + 重试 + 补偿）**

可选实现：

- Outbox 本地消息表（订单写库与消息记录同事务）
- 消息投递失败自动重试
- 多次失败进入死信队列（DLQ）人工补偿

### 15.1 限流

位置：

- Nginx 限流（IP级别）

常用算法：

- 令牌桶
- 漏桶
- 滑动窗口

---

### 15.2 熔断/降级

下游服务超时：

- 直接熔断

框架：

- Sentinel

必须具备：

- 链路追踪：SkyWalking / Jaeger

# 17. 项目总结（面试一句话）

高并发订单系统的关键能力：

- Redis 预扣库存防超卖
- MQ 削峰填谷 + 异步化
- 幂等机制防重复下单
- 延迟消息实现超时关单
- 支付回调幂等保证一致性
- 分库分表支撑订单海量增长
- 限流熔断保障系统稳定
