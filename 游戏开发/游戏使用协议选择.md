在棋牌游戏（大厅 + 房间 + 斗地主/麻将）里，一般是：

**HTTP 用于“低频、一次性、请求-响应”业务**
**WebSocket 用于“实时、频繁、状态同步、推送”业务**

下面按模块给你划分。

---

# ✅ 适合用 HTTP 的（REST 接口）

## 1）账号相关（低频）

* 登录 / 注册
* 获取用户信息
* 刷新 token
* 退出登录

## 2）大厅数据（进入大厅时拉一次）

* 获取房间列表 / 匹配列表
* 获取公告、活动、排行榜
* 获取背包/道具
* 获取商城商品列表
* 获取好友列表

## 3）支付相关（必须 HTTP）

* 下单
* 支付回调
* 查询订单状态

## 4）配置/静态数据

* 拉取游戏配置（倍率、规则、房间限制）
* 拉取版本号、热更新配置

## 5）战绩/记录查询

* 最近对局列表
* 对局详情
* 录像回放数据（也可以下载文件）

📌 总结：HTTP 适合“进入模块时拉数据”的行为。

---

# ✅ 适合用 WebSocket 的（实时通信）

## 1）房间相关（强实时）

* 创建房间/加入房间（也可以 HTTP，但一般直接走 WS）
* 房间人数变化推送
* 玩家准备/取消准备
* 房主解散房间投票
* 房间倒计时同步
* 断线重连状态恢复（snapshot）

## 2）游戏对局逻辑（核心必须 WS）

斗地主：

* 发牌
* 抢地主 / 叫分
* 出牌
* 过牌
* 牌型校验结果同步
* 结算结果推送
* 轮到谁出牌通知

麻将：

* 摸牌/出牌
* 吃碰杠胡
* 听牌提示
* 牌墙剩余
* 结算同步

## 3）实时推送类

* 心跳 ping/pong
* 踢人/封号通知
* 系统广播
* 匹配成功通知
* 聊天消息

📌 总结：只要涉及“多人同步”和“服务器主动推送”，就用 WS。

---

# ⭐ 最佳实践（真实项目常用）

## 登录用 HTTP + 游戏用 WebSocket

典型流程：

1. HTTP 登录 → 拿 token
2. 进入大厅 HTTP 拉数据
3. 建立 WebSocket（带 token）
4. 后面房间/对局全部 WS
5. 战绩查询仍然 HTTP

---

# ⚠️ 哪些功能两者都能做？（建议怎么选）

## 创建房间 / 加入房间

* 可以 HTTP
* 但推荐 **WebSocket**（加入后立刻推送房间状态）

## 匹配

* 匹配请求可以 HTTP
* 但匹配成功必须 WS 推送
* 所以推荐全程 WS（更统一）

---

# ✅ 一句话结论

* **HTTP：登录、支付、配置、排行榜、战绩、商城**
* **WebSocket：房间、匹配、对局、聊天、推送、断线重连、心跳**
