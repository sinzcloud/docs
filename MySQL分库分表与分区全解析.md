# MySQL分库分表与分区全解析

## 一、概念对比（面试必考）

| 维度 | 分区（Partition） | 分表（Sharding） | 分库（Database Sharding） |
|------|------------------|------------------|---------------------------|
| **本质** | 一张逻辑表，多物理文件 | 多张物理表，逻辑合并 | 多个数据库实例 |
| **对应用透明** | ✅ 完全透明，SQL不变 | ❌ 需路由逻辑 | ❌ 需数据源路由 |
| **跨区查询** | ✅ 支持（自动合并） | ❌ 需应用层聚合 | ❌ 需应用层聚合 |
| **事务支持** | ✅ 完整支持 | ❌ 分布式事务 | ❌ 分布式事务 |
| **连接数** | 不增加 | 增加（多表连接） | 大量增加 |
| **单表上限** | 突破OS文件限制 | 突破MySQL表限制 | 突破单库容量 |
| **实现成本** | 低（数据库内置） | 中（中间件/应用层） | 高（中间件+运维） |

## 二、核心区别详解

### 2.1 分区（Partition）

**一张大表 → 多个物理文件，逻辑上仍是一张表**

```sql
-- 建表时定义分区
CREATE TABLE orders (
    id BIGINT,
    order_no VARCHAR(32),
    amount DECIMAL(10,2),
    create_time DATETIME,
    PRIMARY KEY (id, create_time)
) PARTITION BY RANGE (YEAR(create_time)) (
    PARTITION p2022 VALUES LESS THAN (2023),
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025)
);

-- 应用层SQL完全不变，MySQL自动路由到对应分区
SELECT * FROM orders WHERE create_time BETWEEN '2024-01-01' AND '2024-12-31';
-- MySQL只扫描p2024分区，其他分区不扫描
```

**物理存储**：
```
/var/lib/mysql/mydb/
├── orders.frm          # 表结构（一张表）
├── orders#P#p2022.ibd  # p2022分区数据文件
├── orders#P#p2023.ibd  # p2023分区数据文件
└── orders#P#p2024.ibd  # p2024分区数据文件
```

### 2.2 分表（Sharding）

**一张逻辑表 → 多张物理表**

```sql
-- 手动创建多张表
CREATE TABLE orders_0 (
    id BIGINT PRIMARY KEY,
    order_no VARCHAR(32),
    amount DECIMAL(10,2),
    create_time DATETIME
);

CREATE TABLE orders_1 (
    id BIGINT PRIMARY KEY,
    order_no VARCHAR(32),
    amount DECIMAL(10,2),
    create_time DATETIME
);

-- 应用层需要路由逻辑
-- 根据user_id % 2 决定写入 orders_0 还是 orders_1
```

**物理存储**：
```
/var/lib/mysql/mydb/
├── orders_0.frm
├── orders_0.ibd
├── orders_1.frm
└── orders_1.ibd
```

## 三、适用场景对比

### 3.1 什么时候用分区？

| 场景 | 说明 | 示例 |
|------|------|------|
| **日志/流水数据** | 按时间淘汰旧数据 | 保留最近3个月日志 |
| **大表查询有时间范围** | 查询总是带时间条件 | 订单查询按日期 |
| **数据清理需求频繁** | DELETE慢，DROP分区快 | 删除一个月前的数据 |
| **单表数据量大但索引优化已到极限** | >500万行 | 千万级订单表 |
| **希望应用层无感知** | 不想改代码 | 老系统改造 |

### 3.2 什么时候用分表？

| 场景 | 说明 | 示例 |
|------|------|------|
| **单表超过2000万行** | MySQL性能下降明显 | 用户行为表 |
| **写入瓶颈** | 单表写入锁竞争 | 计数器表、消息表 |
| **索引维护成本高** | 索引超过内存 | 大表索引重建慢 |
| **按业务垂直拆分** | 字段太多，冷热分离 | 商品表拆分为商品主表+详情表 |
| **已有分库中间件** | 公司已有Sharding架构 | ShardingSphere |

### 3.3 什么时候两者结合？

```sql
-- 先分库（按用户ID），再分表（按时间分区）
-- 分库分表：32个库 × 64张表 × 按月分区
-- 这是大厂的终极方案
```

## 四、分表方案详解

### 4.1 分表策略

```java
// 1. 哈希取模（最常用）
int tableIndex = userId % 64;  // 0-63
String tableName = "user_" + tableIndex;

// 2. 范围分表（适合自增ID）
// user_0: id 1-100万
// user_1: id 100万-200万
// user_2: id 200万-300万

// 3. 一致性哈希（扩容友好）
// 使用一致性哈希算法，减少扩容时的数据迁移

// 4. 日期分表
// orders_202401, orders_202402, orders_202403
```

### 4.2 分表中间件对比

| 中间件 | 公司 | 优点 | 缺点 |
|--------|------|------|------|
| **ShardingSphere** | Apache | 生态完善，支持5.7/8.0 | 学习成本高 |
| **MyCAT** | 开源 | 配置简单 | 社区活跃度下降 |
| **Vitess** | YouTube | 支持大规模集群 | 部署复杂 |
| **TDDL** | 阿里巴巴 | 淘宝验证 | 开源不活跃 |

### 4.3 分表后的问题

```sql
-- 问题1：跨分片查询
-- 需要查询user_0到user_63，应用层聚合
SELECT * FROM user_0 WHERE age > 18
UNION ALL
SELECT * FROM user_1 WHERE age > 18
-- ... 64张表

-- 问题2：全局ID生成
-- 不能用自增ID，需要分布式ID
-- 解决方案：雪花算法（Snowflake）、美团Leaf、百度UidGenerator

-- 问题3：分页排序
-- 需要从每个分片取数据，应用层再排序
-- 性能极差，通常禁止跨分片分页

-- 问题4：分布式事务
-- 跨库更新需要分布式事务（TCC、Seata）
-- 性能损耗大，尽量避免
```

## 六、选择决策树

```
开始
  │
  ▼
单表数据量多大？
  │
  ├── < 500万行 ──► 不需要分表，优化索引即可
  │
  ├── 500万-2000万行 ──► 考虑分区
  │     │
  │     ├── 查询总带时间范围？ ──► 按时间分区
  │     └── 其他情况 ──► 分区或分表皆可
  │
  └── > 2000万行 ──► 需要分表
        │
        ├── 可以接受改造？ ──► 分表 + 中间件
        ├── 不想改代码？ ──► 分区（但数据量太大效果有限）
        └── 未来会更大？ ──► 分库分表
```

## 八、大厂实践案例

### 案例1：支付宝交易记录
- **策略**：按用户ID分库分表（256库 × 1024表）
- **中间件**：自研分布式数据库
- **分区**：每个分表内部按时间分区
- **结果**：支持万亿级数据

### 案例2：美团订单
- **策略**：按用户ID哈希分库分表
- **中间件**：ShardingSphere + 自研
- **结果**：单表不超过2000万，查询99.9%在1秒内

### 案例3：B站日志
- **策略**：按天分区（不分表）
- **中间件**：无
- **保留周期**：30天自动删除
- **结果**：单表每天1亿数据，分区后查询正常

## 九、常见误区

| 误区 | 真相 |
|------|------|
| 分区能提升所有查询性能 | 只有查询带分区字段才能裁剪 |
| 分表后就能无限扩展 | 跨分片查询、分布式事务是瓶颈 |
| 分库分表是银弹 | 增加系统复杂度，非必要不使用 |
| 分区和分表可以随意切换 | 表结构设计时就决定了，后期改造成本高 |

## 十、总结一句话

> **分区是数据库内置的"垂直切割"，对应用透明，适合按时间淘汰数据的场景；分表是应用层的"水平拆分"，解决单表性能上限，但需要改造代码。两者可以结合使用，分区解决时间维度，分表解决数据量维度。**

**选择建议**：
1. 优先考虑**分区**（成本低、无侵入）
2. 分区不够再用**分表**（中间件方案）
3. 最后才考虑**分库分表**（终极方案，慎用）
